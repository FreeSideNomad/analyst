# Plan — Feature 005: Relational database federation

> Checkpoint 4 (autonomy high — no human gate; decisions recorded here).

## Architecture decisions (researched 2026-07-03)

| Engine | Path | Rationale |
|---|---|---|
| SQLite | DuckDB `ATTACH … (TYPE sqlite, READ_ONLY)` (core `sqlite` extension); stdlib-`sqlite3` bridge fallback if the extension can't load (offline box) | Core scanner, solid; deterministic CI base |
| PostgreSQL | DuckDB `ATTACH … (TYPE postgres, READ_ONLY)` (core `postgres` extension); declared keys via `postgres_query()` push-down on `information_schema` | Core scanner, solid |
| SQL Server | **Driver bridge** on `pymssql` (installs clean on macOS arm / py3.14) with query push-down | The community `mssql` DuckDB extension exists (hugr-lab, native TDS) but self-describes as **experimental**; the bridge is code we control and unit-test offline |
| IBM DB2 | **Driver bridge** on `ibm_db` (`ibm_db_dbi` DB-API); keys via `SYSCAT` | No DuckDB scanner exists for DB2 |

Both paths sit behind one `Connector` protocol in `engine/federation.py`.
Nothing is bulk-copied: the ATTACH path queries through the scanner; the
bridge path pushes profiling SQL down to the source and only aggregates,
samples and capped result sets come back (registered into DuckDB via
`register` when needed). Heavy drivers live in a `dbs` optional-dependency
group so default `uv sync` stays lean; unit tests exercise the bridge through
fake drivers + the stdlib-sqlite driver.

## Layers

- **domain/connection.py** (new, pure): `DatabaseEngine`, `ConnectionSpec`
  (holds the secret; validation), `ConnectionSummary` (secret-free),
  `ForeignKey`/`TableKeys`, and `catalog_for_table(...)` — deterministic
  plain-English `CatalogEntry` from profile + declared keys (no LLM).
- **engine/federation.py** (new): `FederationError`, `Connector` protocol,
  `DuckDBAttachConnector` (sqlite/postgres, own in-memory DuckDB con per
  connection), `BridgeConnector` + per-engine dialects (mssql/db2/sqlite-
  fallback), `create_connector` factory, `FederationService` (connect /
  list / tables / fetch / detach registry). Profiling: ATTACH path reuses the
  feature-001 `profile_relation` over a temp view; bridge path pushes
  profiling SQL down (same `DatasetProfile` shape, quantiles omitted).
- **api/routes/databases.py** (new): wire schemas (no password field on any
  response model — leak-proof by construction), `DatabaseManager` holder that
  follows the repo instance (so `/api/_reset` also resets connections),
  `POST /api/databases/connect`, `GET /api/databases`,
  `DELETE /api/databases/{name}`. Connected tables land in the repository as
  `DatasetRecord`s named `<connection>.<table>`.
- **api/repository.py** (shared, minimal): `add_records` / `remove_records`
  hooks on the protocol + both implementations.
- **api/app.py** (shared, minimal): include the databases router.
- **frontend**: new `api/databases.ts` (types + calls), new
  `components/DatabasePanel.tsx` (Databases section: list, detach, connect
  form with labelled fields), `stores/catalog-store.ts` (shared: connections
  state + connect/detach actions), `pages/WorkspacePage.tsx` (shared: replace
  the "Connect a database — soon" placeholder with `<DatabasePanel />`).

## Fixtures & tests

- Bundled sample DB: `tests/golden/chinook.sqlite` — a small subset of the
  MIT-licensed Chinook DB (full schema + declared PK/FKs, reduced rows),
  generated by `tests/golden/make_chinook.py` (downloads + subsets; cached).
  Backs unit tests, acceptance scenarios and the UI e2e — offline-safe.
- Acceptance: `acceptance/e2e_005.py` on `e2e_base.py`;
  `features/005-db-federation/.handlers` → `acceptance.e2e_005`.
- Unit: domain validation, connector paths (real sqlite ATTACH, bridge over
  stdlib sqlite + fake mssql/db2 drivers, no-copy invariant), API routes.
- Live (Docker, `-m live`): `tests/live/test_federation_dbs.py` against
  `docker-compose.dbs.yml` (Postgres+Pagila, SQL Server+Northwind, DB2
  SAMPLE) via `make dbs-up` / `dbs-down`. Apple-silicon caveats → runbook.
